# 设计文档：LLM Code Agent

## 原始需求

我有一个C语言数据库项目，我想要编写基于LLM的code agent，使得用户可以输入特性描述，code agent能够独立完成开发。
目前我的思路是：
先进行预处理（不调LLM）：
1.	解析项目的文件得到摘要，这个摘要包括函数的签名、功能和流程。
2.	得到关键流程的函数调用链，例如写入数据、查询、后台任务等。
3.	数据库使用的技术，例如火山执行引擎、Columbia优化器。
然后
1.	调LLM api，传入上述预处理的信息，使得LLM能够理解项目结构
2.	调LLM api，描述需要开发的特性，获知需要新增或者修改的文件以及函数名称
3.	调LLM api，根据文件和函数名称传入内容，获知修改的点



## 背景与目标

为 C 语言数据库项目构建 LLM-based code agent：用户输入特性描述，agent 独立完成代码修改。

**核心约束：**
- 在线阶段（每次特性请求）LLM 调用次数尽量少（目标 2 次）
- 预处理阶段可批量调用 LLM（一次性执行，结果缓存）
- 环境不支持 ctags/clang 等工具，调用链由 Python 静态分析构建

## 整体架构

```
┌─────────────────────────────────────────────────────┐
│ PHASE 1: 预处理（一次性执行，结果持久化缓存）         │
│                                                      │
│  Step 1: Python 静态分析                             │
│    ├─ 正则 + 括号匹配提取函数定义                    │
│    ├─ 从函数体中提取调用关系                         │
│    └─ BFS 从用户指定入口构建调用链                   │
│                                                      │
│  Step 2: LLM 批量生成富摘要（40 函数/批）            │
│    └─ 函数摘要包含：功能、所属子系统、调用场景       │
│                                                      │
│  Step 3: 构建多级索引                                │
│    ├─ 子系统映射（LLM：目录树 → 语义标签）           │
│    ├─ 文件级摘要（聚合函数摘要，无 LLM）             │
│    └─ 函数级富摘要（最终检索目标）                   │
└─────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────┐
│ PHASE 2: 在线处理（每次特性请求，2-3 次 LLM 调用）   │
│                                                      │
│  检索层（0 LLM，纯计算）:                            │
│    Layer 1: 子系统映射 → 定位候选目录                │
│    Layer 2: BM25 限域检索 → top-25 候选函数          │
│    Layer 3: 调用链扩展 → 补充结构性相关函数           │
│                                                      │
│  Call 1: 规划（~30 函数摘要）→ 实现计划 + 文件列表   │
│  Call 2: 实现（受影响文件完整内容）→ unified diff    │
│  Call 3: （仅大型特性）接口整合 → 头文件 diff        │
└─────────────────────────────────────────────────────┘
```

## Phase 1：预处理详细设计

### Step 1：Python 静态分析 C 源码

**函数提取**（`preprocess/c_parser.py`）：

- 用正则匹配函数定义头部（返回类型 + 函数名 + 参数列表）
- 从 `{` 开始做括号深度计数，精确定位函数体边界
- 从函数体内提取所有 `identifier(` 模式的调用，过滤 C 关键字

**关键设计：** 正则结尾用 `(?=[\s{])` lookahead 而非 `\)\s*$`，支持 `) {` 同行的常见 C 风格。

**调用链构建**（`preprocess/callchain_builder.py`）：

- 用户在 `callchain_entries.yaml` 中指定入口函数
- BFS 遍历构建嵌套调用树，最大深度 8 层
- 已展开的函数用 `__ref__` 标记，避免重复展开

### Step 2：LLM 批量生成富摘要

**摘要质量是检索链的上限。** 函数名或简单描述不足以支撑高质量检索，摘要必须包含：

| 字段 | 作用 |
|------|------|
| `summary` | 功能描述（≤40字），描述目的而非实现 |
| `subsystem` | 所属子系统（storage/executor/optimizer/...）|
| `scenario` | 被谁调用、在什么场景下使用（≤60字）|
| `data_structures` | 涉及的核心结构体或数据库概念 |

**差摘要 vs 好摘要示例：**

```
差：分配一个新的缓冲帧
好：缓冲池核心分配函数，存储/缓冲子系统。被 heap_fetch、index_scan 调用，
    与 WAL 刷盘和 checkpoint 协作确保持久性。
```

好摘要在语义和词汇两个维度大幅提升检索召回率。

**批处理策略：** 每批 40 个函数，动态控制约 8k tokens/批。每批传入函数签名 + 函数体 + 已知调用者，使 LLM 能推断调用场景。

### Step 3：多级索引

三层结构解决"BM25 遗漏整个文件"问题：

```
子系统层（目录）  →  文件层（.c 文件）  →  函数层（函数）
                                            ↑
                                        最终检索目标
```

- **子系统映射**：LLM 一次性分析项目目录树，生成 `{目录: {关键词列表}}` 的语义标签
- **文件层**：由函数摘要聚合生成，无额外 LLM 调用
- **函数层**：Step 2 的输出，是检索的直接对象

## Phase 2：在线处理详细设计

### 检索层：为何不直接用 BM25？

| 方案 | 问题 |
|------|------|
| 全库 BM25 top-80 | 可能系统性遗漏某个模块的所有函数 |
| 全库 Embedding | 语义相似 ≠ 代码结构相关；需额外基础设施 |
| **三层融合（本方案）** | 先缩小范围，再检索，再补充结构性关联 |

### Layer 1：子系统定位（0 LLM）

```
特性描述 → 与子系统关键词做重叠匹配 → 候选目录列表
```

- 将候选空间从 ~5000 函数缩小到 ~200-500 函数（同一子系统内）
- 无匹配时自动回退到全库（保证 recall）

### Layer 2：限域 BM25（0 LLM）

在 Layer 1 确定的子系统内做 BM25：

- 中英文混合分词（英文按词、中文按字 + 双字 n-gram）
- 文档权重：summary 和 scenario 字段重复两次，提升语义字段权重
- 返回 top-25 个候选函数

### Layer 3：调用链扩展（0 LLM）

对 BM25 top-5 种子函数展开：

- **向下扩展（depth=2）**：找到种子调用的函数，用于发现实现细节
- **向上扩展（depth=1）**：找到调用种子的函数，用于评估影响范围
- **黑名单剪枝**：跳过 `palloc`/`elog`/`memcpy` 等工具函数，避免结果爆炸

**最终候选集**：~30-40 个函数摘要，传入 Call 1。

### Call 1：规划

**输入（token 预算）：**
- System: `project_overview.md` + 调用链摘要
- User: 特性描述 + ~30 个候选函数（每个约 3 行文本）

**输出（JSON）：**
```json
{
  "complexity": "medium|high",
  "affected_files": [...],
  "functions_to_modify": [...],
  "functions_to_add": [{"name": "...", "in_file": "..."}],
  "new_files": [...],
  "header_changes": [...],
  "implementation_plan": "分步骤说明..."
}
```

### Call 2：实现

- 传入 `affected_files` 的完整文件内容
- 强制输出 unified diff，不输出解释文字
- diff 可直接用 `patch -p1` 应用

### Call 3（仅 `complexity=high`）

- 检查头文件声明与跨模块接口一致性
- 只输出头文件的 diff，不重复 Call 2 的内容

## 关键设计决策

### 为什么用 BM25 而非 Embedding？

在已有调用链的前提下，BM25 + 调用链扩展的组合优于纯 Embedding：
- 调用链能发现词汇上完全不相关但结构上耦合的函数
- BM25 对精确函数名/API 名匹配效果好
- 不需要额外的 embedding 基础设施

可以后续在 Layer 2 引入 Embedding（Hybrid Search = BM25 + Embedding），但在子系统已限域的前提下收益边际递减。

### 为什么摘要字段需要"调用场景"？

代码检索的核心问题：用户用业务语言描述（"支持嵌套事务"），代码用技术标识符实现（`savepoint_push()`）。

词汇鸿沟只能通过摘要中的场景描述来弥合：
- 好的 scenario 字段包含"被 exec_insert 调用"
- 用户的"写入"描述就能通过 `exec_insert` → `heap_insert` 这条语义链找到目标

### 增量更新策略

- 用 SHA-256 文件哈希跟踪变更
- 只重新处理 hash 变化的文件
- 需要重建完整调用图（部分调用图更新会引入不一致）
- 子系统映射在项目目录结构不变时不需要重新生成（`--regen-subsystem` 手动触发）

## LLM 调用成本估算

以 10 万行、~2500 个函数的中型项目为例：

| 阶段 | 调用次数 | 模型 | 估算成本 |
|------|---------|------|---------|
| 预处理-函数摘要 | ~63 批（2500÷40）| Haiku | ~$0.5 |
| 预处理-子系统映射 | ~2 次 | Haiku | <$0.01 |
| 在线-每次特性 | 2-3 次 | Sonnet | ~$0.05-0.15 |

预处理为一次性成本，在线处理每次特性请求成本极低。
